# スペック駆動開発 (SDD)

## パワーの反転

数十年にわたり、コードが王様でした。仕様書はコードをサポートするものであり、コーディングという「本物の仕事」が始まった後に構築して破棄する足場でした。私たちはPRDを書いて開発を導き、設計ドキュメントを作成して実装を通知し、図を描いてアーキテクチャを可視化しました。しかし、これらは常にコード自体に従属するものでした。コードが真実でした。他のすべては、せいぜい良い意図にすぎませんでした。コードが真実の源であり、それが進むにつれて、仕様は追いつくことがめったにありませんでした。資産（コード）と実装が一つであるため、コードからビルドしようとする without で並列実装を持つことは容易ではありません。

スペック駆動開発（SDD）はこの権力構造を逆転させます。仕様書はコードをサポートしません—コードが仕様書をサポートします。製品要件定義（PRD）は実装のガイドではありません。実装を生成するソースです。技術計画はコーディングを通知するドキュメントではありません。コードを生成する正確な定義です。これはソフトウェアの構築方法への段階的な改善ではありません。開発を駆動するものについての根本的な再考です。

仕様と実装のギャップは、ソフトウェア開発の始まり以来、その問題を引きずってきました。より良いドキュメント、より詳細な要件、より厳格なプロセスでそれを橋渡ししようとしました。これらのアプローチはギャップを必然として受け入れるため失敗します。ギャップを狭めようとするだけで、決して排除しません。SDDは仕様とその具体的な実装計画が実行可能であることでギャップを排除します。仕様と実装計画がコードを生成するとき、ギャップはありません—変換だけです。

この変換が可能になったのは、AIが複雑な仕様を理解し、実装でき、詳細な実装計画を作成できるようになったからです。しかし、構造のない生のAI生成は混沌を生みます。SDDは、仕様とその後の実装計画が、作業システムを生成するのに十分に正確で完全で曖昧でないことで、その構造を提供します。仕様が主要なアーティファクトになります。コードは（実装計画からの実装として）特定の言語とフレームワークでのその表現になります。

この新しい世界では、ソフトウェアのメンテナンスは仕様の進化を意味します。開発チームの意図は自然言語（「**意図駆動開発**」）、設計アセット、コア原則、その他のガイドラインで表現されます。開発の**共通言語**はより高いレベルに移り、コードはラストマイルのアプローチになります。

デバッグは、間違ったコードを生成する仕様とその実装計画を修正することを意味します。リファクタリングは明確さのために再構造化することを意味します。開発ワークフロー全体が、実装計画とコードを継続的に再生成される出力として、中央の真実源として仕様を中心に再編成されます。新しい機能でアプリを更新するか、私たちは創造的な存在であるため、新しい並列実装を作成する means は、仕様を再訪し、新しい実装計画を作成することです。このプロセスはしたがって0 -> 1、(1', ..)、2、3、Nです。

開発チームは、その創造性、実験、批判的思考に焦点を当てます。

## 実践でのSDDワークフロー

ワークフローはアイデアから始まります—しばしば曖昧で不完全です。AIとの反復対話を通じて、このアイデアは包括的なPRDになります。AIは明確化の質問をし、エッジケースを特定し、正確な受け入れ基準を定義するのを助けます。伝統的な開発で数日の会議とドキュメント作成が必要なものが、集中した仕様作業の数時間で起こります。これは伝統的なSDLCを変換します—要件と設計が離散的なフェーズではなく、連続的な活動になります。これは**チームプロセス**をサポートし、チームレビューされた仕様が表現され、バージョン管理され、ブランチで作成され、マージされます。

製品マネージャーが受け入れ基準を更新すると、実装計画は自動的に影響を受ける技術的決定をフラグ付けします。アーキテクトがより良いパターンを発見すると、PRDは新しい可能性を反映するように更新されます。

この仕様プロセス全体で、研究エージェントが重要なコンテキストを収集します。彼らはライブラリの互換性、パフォーマンスベンチマーク、セキュリティの影響を調査します。組織の制約が検出され、自動的に適用されます—あなたの会社のデータベース標準、認証要件、デプロイメントポリシーがシームレスにすべての仕様に統合されます。

PRDから、AIは要件を技術的決定にマッピングする実装計画を生成します。すべての技術的選択には文書化された根拠があります。すべてのアーキテクチャ的決定は特定の要件にさかのぼります。このプロセス全体で、一貫性の検証が継続的に品質を向上させます。AIは仕様の曖昧さ、矛盾、ギャップを分析します—一度きりのゲートではなく、継続的な洗練としてです。

コード生成は、仕様とその実装計画が十分に安定した時点で始まりますが、「完全」である必要はありません。早期の生成は探索的かもしれません—仕様が実際の実装で意味をなすかどうかをテストします。ドメインの概念がデータモデルになります。ユーザーストーリーがAPIエンドポイントになります。受け入れシナリオがテストになります。これは仕様を通じて開発とテストを統合します—テストシナリオはコードの後に書かれるのではなく、実装とテストの両方を生成する仕様の一部です。

フィードバックループは初期開発を超えて拡張されます。本番のメトリクスとインシデントは、ホットフィックスをトリガーするだけでなく、次の再生成のための仕様を更新します。パフォーマンスのボトルネックが新しい非機能要件になります。セキュリティの脆弱性が将来のすべての生成に影響を与える制約になります。仕様、実装、運用現実の間のこの反復的なダンスが真の理解が現れ、伝統的なSDLCが連続的な進化に変換される場所です。

## なぜ今SDDが重要なのか

3つのトレンドがSDDを可能にするだけでなく、必要としています：

まず、AIの能力が自然言語の仕様が信頼性高く作業コードを生成できるしきい値に達しました。これは開発者を置き換えることについてではありません—仕様から実装への機械的な翻訳を自動化することで、彼らの有効性を増幅することについてです。これは探索と創造性を増幅し、「やり直し」を容易にし、追加、減算、批判的思考をサポートします。

第二に、ソフトウェアの複雑さが指数関数的に成長し続けています。現代のシステムは数十のサービス、フレームワーク、依存関係を統合します。これらのすべてのピースを手動プロセスで元の意図と整列させることはますます困難になります。SDDは仕様駆動の生成を通じて体系的な整列を提供します。フレームワークはAIファーストのサポートを提供するために進化するかもしれません、人間ファーストのサポートではなく、または再利用可能なコンポーネントを中心にアーキテクトします。

第三に、変化のペースが加速しています。今日の要件変更はこれまで以上に急速に変化します。方向転換はもはや例外的ではありません—期待されています。現代の製品開発は、ユーザーのフィードバック、市場条件、競争圧力に基づく迅速な反復を要求します。伝統的な開発はこれらの変更を障害として扱います。各方向転換は手動でドキュメント、設計、コードに変更を伝播する必要があります。結果は、速度を制限する遅く慎重な更新、または技術的負債を蓄積する速く無謀な変更のいずれかです。

SDDは、もし/シミュレーション実験をサポートできます：「もしビジネスニーズを満たすためにアプリケーションを再実装または変更してより多くのTシャツを販売する必要がある場合、どのように実装して実験しますか？」

SDDは要件変更を障害から通常のワークフローに変換します。仕様が実装を駆動するとき、方向転換は手動の書き換えではなく体系的な再生成になります。PRDのコア要件を変更し、影響を受ける実装計画が自動的に更新されます。ユーザーストーリーを変更し、対応するAPIエンドポイントが再生成されます。これは初期開発についてだけでなく、避けられない変更を通じてエンジニアリングの速度を維持することについてです。

## コア原則

**共通言語としての仕様**：仕様が主要なアーティファクトになります。コードが特定の言語とフレームワークでのその表現になります。ソフトウェアのメンテナンスは仕様の進化を意味します。

**実行可能な仕様**：仕様は作業システムを生成するのに十分に正確で完全で曖昧でない必要があります。これは意図と実装のギャップを排除します。

**継続的な洗練**：一貫性の検証は一度きりのゲートではなく、継続的に起こります。AIは仕様の曖昧さ、矛盾、ギャップを継続的なプロセスとして分析します。

**研究駆動のコンテキスト**：研究エージェントが仕様プロセス全体で重要なコンテキストを収集し、技術的オプション、パフォーマンスの影響、組織の制約を調査します。

**双方向のフィードバック**：本番の現実が仕様の進化を通知します。メトリクス、インシデント、運用の学習が仕様の洗練の入力になります。

**探索のためのブランチ**：同じ仕様から複数の実装アプローチを生成して、異なる最適化ターゲット（パフォーマンス、保守性、ユーザーエクスペリエンス、コスト）を探索します。

## 実装アプローチ

今日、SDDを実践するには、既存のツールを組み立て、プロセス全体で規律を維持する必要があります。このメソッドは以下を使用して実践できます：

- 繰り返しの仕様開発のためのAIアシスタント
- 技術的コンテキストを収集するための研究エージェント
- 仕様から実装への翻訳のためのコード生成ツール
- 仕様ファーストのワークフローに適応したバージョン管理システム
- 仕様ドキュメントのAI分析を通じた一貫性チェック

鍵は、コードを生成された出力として、仕様を真実の源として扱い、コードが仕様をサポートするのではなく、その逆になることです。

## コマンドによるSDDの合理化

SDDメソッドは、仕様 → 計画 → タスクのワークフローを自動化する3つの強力なコマンドを通じて大幅に強化されます：

### `/speckit.specify` コマンド

このコマンドは、単純な機能説明（ユーザープロンプト）を自動リポジトリ管理を伴う完全な構造化仕様に変換します：

1. **自動機能番号付け**：既存の仕様をスキャンして次の機能番号（例：001、002、003、…、1000 — 自動的に3桁以上に拡張）を決定します
2. **ブランチ作成**：説明から意味のあるブランチ名を生成し、自動的に作成します
3. **テンプレートベースの生成**：要件で機能仕様テンプレートをコピーしてカスタマイズします
4. **ディレクトリ構造**：関連ドキュメントのための適切な `specs/[branch-name]/` 構造を作成します

### `/speckit.plan` コマンド

機能仕様が存在するとき、このコマンドは包括的な実装計画を作成します：

1. **仕様分析**：機能要件、ユーザーストーリー、受け入れ基準を読み取って理解します
2. **憲法準拠**：プロジェクト憲法とアーキテクチャ原則との整合性を確保します
3. **技術的翻訳**：ビジネス要件を技術アーキテクチャと実装の詳細に変換します
4. **詳細なドキュメント**：データモデル、API契約、テストシナリオのためのサポートドキュメントを生成します
5. **クイックスタート検証**：主要な検証シナリオをキャプチャするクイックスタートガイドを生成します

### `/speckit.tasks` コマンド

計画が作成された後、このコマンドは計画と関連する設計ドキュメントを分析して実行可能なタスクリストを生成します：

1. **入力**：`plan.md`（必須）と、存在する場合は `data-model.md`、`contracts/`、`research.md` を読み取ります
2. **タスク導出**：契約、エンティティ、シナリオを特定のタスクに変換します
3. **並列化**：独立したタスクに `[P]` をマークし、安全な並列グループを概説します
4. **出力**：タスクエージェントによる実行の準備が整った機能ディレクトリに `tasks.md` を書き込みます

### 例：チャット機能の構築

これらのコマンドが伝統的な開発ワークフローをどのように変換するかの例：

**伝統的なアプローチ：**

```text
1. ドキュメントにPRDを書く（2-3時間）
2. 設計ドキュメントを作成する（2-3時間）
3. 手動でプロジェクト構造をセットアップする（30分）
4. 技術仕様を書く（3-4時間）
5. テスト計画を作成する（2時間）
合計：約12時間のドキュメント作業
```

**コマンドを使用したSDDアプローチ：**

```bash
# ステップ1：機能仕様を作成する（5分）
/speckit.specify メッセージ履歴とユーザー存在を備えたリアルタイムチャットシステム

# これは自動的に：
# - ブランチ "003-chat-system" を作成
# - specs/003-chat-system/spec.md を生成
# - 構造化された要件でそれを埋める

# ステップ2：実装計画を生成する（5分）
/speckit.plan リアルタイムメッセージングのWebSocket、履歴のPostgreSQL、存在のRedis

# ステップ3：実行可能なタスクを生成する（5分）
/speckit.tasks

# これは自動的に作成します：
# - specs/003-chat-system/plan.md
# - specs/003-chat-system/research.md（WebSocketライブラリの比較）
# - specs/003-chat-system/data-model.md（MessageとUserスキーマ）
# - specs/003-chat-system/contracts/（WebSocketイベント、RESTエンドポイント）
# - specs/003-chat-system/quickstart.md（主要な検証シナリオ）
# - specs/003-chat-system/tasks.md（計画から導出されたタスクリスト）
```

15分で、あなたは持っています：

- ユーザーストーリーと受け入れ基準を備えた完全な機能仕様
- 技術的選択と根拠を備えた詳細な実装計画
- コード生成のための準備が整ったAPI契約とデータモデル
- 自動化と手動テストのための包括的なテストシナリオ
- 機能ブランチで適切にバージョン管理されたすべてのドキュメント

### 構造化された自動化の力

これらのコマンドは時間を節約するだけでなく、一貫性と完全性を強制します：

1. **忘��られた詳細なし**：テンプレートが非機能要件からエラーハンドリングまですべての側面を考慮することを保証します
2. **追跡可能な決定**：すべての技術的選択が特定の要件にリンクされます
3. **生きているドキュメント**：仕様がコードを生成するため、仕様はコードと同期したままになります
4. **迅速な反復**：要件を変更し、数日ではなく数分で計画を再生成します

これらのコマンドは、仕様を静的なドキュメントではなく実行可能なアーティファクトとして扱うことでSDDの原則を具現化します。彼らは仕様プロセスを必要な悪から開発の駆動力に変換します。

### テンプレート駆動の品質：構造がより良い結果のためにLLMを制約する方法

これらのコマンドの真の力は自動化だけでなく、テンプレートがLLMの動作をより高品質の仕様に向かって生産的な方法で導く方法にあります。テンプレートはLLMの出力を生産的な方法で制約する洗練されたプロンプトとして機能します：

#### 1. **早期の実装詳細の防止**

機能仕様テンプレートは明示的に指示します：

```text
- ✅ ユーザーが必要とするものとその理由に焦点を当てる
- ❌ 実装方法を避ける（テクノロジースタック、API、コード構造なし）
```

この制約はLLMが適切な抽象レベルを維持することを強制します。LLMが自然に「ReactとReduxを使用して実装する」にジャンプするとき、テンプレートは「ユーザーがデータのリアルタイム更新を必要とする」に焦点を当て続けます。この分離は、実装技術が変更されても仕様が安定したままであることを保証します。

#### 2. **明示的な不確実性マーカーの強制**

両方のテンプレートは `[NEEDS CLARIFICATION]` マーカーの使用を義務付けます：

```text
ユーザープロンプトからこの仕様を作成するとき：
1. **すべての曖昧性をマークする**：[NEEDS CLARIFICATION: specific question] を使用
2. **推測しない**：プロンプトが何かを指定しない場合、それをマークする
```

これは、可能性は高いが間違っている可能性のある仮定をするLLMの共通の動作を防ぎます。LLMが「ログインシステム」がメール/パスワード認証を使用すると推測する代わりに、[NEEDS CLARIFICATION: auth method not specified - email/password, SSO, OAuth?] としてマークする必要があります。

#### 3. **チェックリストによる構造化された思考**

テンプレートは、仕様の「単位テスト」として機能する包括的なチェックリストを含みます：

```markdown
### 要件の完全性

- [ ] [NEEDS CLARIFICATION] マーカーが残っていない
- [ ] 要件がテスト可能で曖昧でない
- [ ] 成功基準が測定可能
```

これらのチェックリストは、LLMにその出力を体系的に自己レビューさせ、見逃される可能性のあるギャップを捕捉します。LLMに品質保証フレームワークを与えるようなものです。

#### 4. **憲法準拠によるゲート**

実装計画テンプレートはフェーズゲートを通じてアーキテクチャ原則を強制します：

```markdown
### フェーズ -1: 実装前のゲート

#### 単純性ゲート（第VII条）

- [ ] ≤3つのプロジェクトを使用しているか？
- [ ] 未来の証明なし？

#### 反抽象ゲート（第VIII条）

- [ ] フレームワークを直接使用しているか？
- [ ] 単一のモデル表現？

#### 統合ファーストゲート（第IX条）

- [ ] 契約が定義されているか？
- [ ] 契約テストが書かれているか？
```

これらのゲートは、LLMが複雑さを明示的に正当化することを強制することで、過剰なエンジニアリングを防ぎます。ゲートが失敗すると、LLMは「複雑さの追跡」セクションで理由を文書化する必要があります、アーキテクチャ的決定の説明責任を作成します。

#### 5. **階層的な詳細管理**

テンプレートは適切な情報アーキテクチャを強制します：

```text
**重要**：この実装計画は高レベルで読みやすく保つ必要があります。
コードサンプル、詳細なアルゴリズム、または広範な技術仕様は
適切な `implementation-details/` ファイルに配置する必要があります
```

これは、仕様が読みにくいコードのダンプになる共通の問題を防ぎます。LLMは適切な詳細レベルを維持し、主要なドキュメントをナビゲート可能にしながら、複雑さを別々のファイルに抽出することを学びます。

#### 6. **テストファーストの思考**

実装テンプレートはテストファーストの開発を強制します：

```text
### ファイル作成順序
1. `contracts/` をAPI仕様で作成
2. 順序でテストファイルを作成：契約 → 統合 → e2e → 単体
3. テストをパスするためにソースファイルを作成
```

この順序制約は、LLMが実装の前にテストの可視性と契約について考えることを保証し、より堅牢で検証可能な仕様につながります。

#### 7. **仕様外の機能の防止**

テンプレートは明示的に推測を禁じます：

```text
- [ ] 推測的または「必要になるかもしれない」機能なし
- [ ] すべてのフェーズに明確な前提条件と成果物がある
```

これは、LLMが実装を複雑にする「あってもよい」機能を追加することを防ぎます。すべての機能は明確な受け入れ基準を持つ具体的なユーザーストーリーにさかのぼる必要があります。

### 累積効果

これらの制約が一緒に働いて、以下の仕様を生成します：

- **完全性**：チェックリストが何も忘れられないことを保証します
- **曖昧さのない**：強制された明確化マーカーが不確実性を強調します
- **テスト可能**：プロセスに組み込まれたテストファーストの思考
- **保守性**：適切な抽象レベルと情報階層
- **実装可能**：具体的な成果物を持つ明確なフェーズ

テンプレートはLLMを創造的な作家から規律ある仕様エンジニアに変換し、その能力を一貫して高品質の、実行可能な仕様を生成するために導きます、それらが真に開発を駆動します。

## 憲法の基盤：アーキテクチャ的規律の強制

SDDの中心には、仕様がコードになる方法を統治する不変の原則のセットである憲法があります。憲法（`memory/constitution.md`）は、すべての生成された実装が一貫性、単純性、品質を維持することを保証するシステムのアーキテクチャ的DNAとして機能します。

### 開発の九つの条項

憲法は開発プロセスのすべての側面を形作る九つの条項を定義します：

#### 第I条：ライブラリファースト原則

すべての機能は、例外なく、スタンドアロンのライブラリとして始まる必要があります。これは最初からモジュール設計を強制します：

```text
Specifyのすべての機能は、スタンドアロンのライブラリとしてその存在を開始する必要があります。
機能は、再利用可能なライブラリコンポーネントに抽象化される前に、
アプリケーションコード内で直接実装されてはなりませ���。
```

この原則は、仕様が単一のアプリケーションではなく、モジュール化され、再利用可能なコードを生成することを保証します。LLMが実装計画を生成するとき、機能を明確な境界と最小限の依存関係を持つライブラリとして構造化する必要があります。

#### 第II条：CLIインターフェース命令

すべてのライブラリはコマンドラインインターフェースを通じてその機能を公開する必要があります：

```text
すべてのCLIインターフェースは：
- テキストを入力として受け取る（stdin、引数、またはファイルを介して）
- テキストを出力として生成する（stdoutを介して）
- 構造化データ交換のためにJSON形式をサポートする
```

これは観測可能性とテスト可能性を強制します。LLMは機能を不透明なクラス内に隠すことができません—すべてがテキストベースのインターフェースを通じてアクセス可能で検証可能である必要があります。

#### 第III条：テストファーストの義務

最も変革的な条項—テストの前にコードなし：

```text
これはNON-NEGOTIABLEです：すべての実装は厳格なテスト駆動開発に従う必要があります。
実装コードは書かれる前に：
1. 単体テストが書かれる
2. テストが検証され、ユーザーによって承認される
3. テストがFAIL（赤色フェーズ）することを確認される
```

これはAIコード生成の伝統的なアプローチを完全に逆転させます。コードを生成してそれが動作することを願う代わりに、LLMはまず動作を定義する包括的なテストを生成し、それらを承認し、それからのみ実装を生成する必要があります。

#### 第VII条と第VIII条：単純性と反抽象

これらのペアの条項は過剰なエンジニアリングと戦います：

```text
セクション7.3：最小プロジェクト構造
- 初期実装の最大3つのプロジェクト
- 追加のプロジェクトは文書化された根拠を必要とする

セクション8.1：フレームワークの信頼
- フレームワークの機能をラップする代わりに直接使用する
```

LLMが自然に洗練された抽象を作成するとき、これらの条項は複雑さのすべてのレイヤーを正当化することを強制します。実装計画テンプレートの「フェーズ-1ゲート」がこれらの原則を直接強制します。

#### 第IX条：統合ファーストテスト

分離された単体テストよりも現実世界のテストを優先します：

```text
テストは現実的な環境を使用する必要があります：
- モックよりも実際のデータベースを優先
- スタブよりも実際のサービスインスタンスを使用
- 実装の前に契約テストを必須
```

これは生成されたコードが理論だけでなく実際には動作することを保証します。

### テンプレートを通じた憲法の強制

実装計画テンプレートが具体的なチェックポイントを通じてこれらの条項を運用します：

```markdown
### フェーズ -1: 実装前のゲート

#### 単純性ゲート（第VII条）

- [ ] ≤3つのプロジェクトを使用しているか？
- [ ] 未来の証明なし？

#### 反抽象ゲート（第VIII条）

- [ ] フレームワークを直接使用しているか？
- [ ] 単一のモデル表現？

#### 統合ファーストゲート（第IX条）

- [ ] 契約が定義されているか？
- [ ] 契約テストが書かれているか？
```

これらのゲートはアーキテクチャ的原則のコンパイル時チェックとして機能します。LLMはゲートを通過するか、「複雑さの追跡」セクションで正当化された例外を文書化しない限り進むことができません。

### 不変の原則の力

憲法の力はその不変性にあります。実装の詳細は進化できますが、コア原則は一定です。これは以下を提供します：

1. **時間の一致性**：今日生成されたコードは来年生成されたコードと同じ原則に従います
2. **LLM間の一致性**：異なるAIモデルがアーキテクチャ的に互換性のあるコードを生成します
3. **アーキテクチャ的完全性**：すべての機能がシステム設計を強化します
4. **品質保証**：テストファースト、ライブラリファースト、単純性の原則が保守可能なコードを保証します

### 憲法の進化

原則は不変ですが、その適用は進化できます：

```text
セクション4.2：修正プロセス
この憲法の変更には：
- 変更の理由の明示的な文書化
- プロジェクトのメンテナによるレビューと承認
- 後方互換性の評価
```

これはメソッドが安定性を維持しながら学び、改善できることを可能にします。憲法は日付付きの修正でその進化を示し、原則が実際の経験に基づいて洗練される方法を示します。

### ルールを超えて：開発の哲学

憲法は単なるルールブックではありません—LLMがコード生成について考える方法を形作る哲学です：

- **観測可能性 over 不透明性**：すべてがCLIインターフェースを通じて検査可能である必要があります
- **巧妙さ over 単純性**：単純に始め、証明された必要性がある場合にのみ複雑さを追加
- **分離 over 統合**：人工的なものではなく、現実的な環境でテスト
- **モノリス over モジュール性**：すべての機能は明確な境界を持つライブラリです

これらの原則を仕様と計画プロセスに組み込むことで、SDDは生成されたコードが機能的であるだけでなく、保守可能でテスト可能でアーキテクチャ的に健全であることを保証します。憲法はAIをコードジェネレータからアーキテクチャのパートナーに変換し、システム設計原則を尊重し強化します。

## 変換

これは開発者を置き換えることや創造性を自動化することについてではありません。それは機械的な翻訳を自動化することで人間の能力を増幅することについてです。仕様、研究、コードが一緒に進化する緊密なフィードバックループを作成することについてです、各反復がより深い理解と意図と実装の間のより良い整列をもたらします。

ソフトウェア開発には、意図と実装の間の整列を維持するためのより良いツールが必要です。SDDは、単にガイドするのではなく、コードを生成する実行可能な仕様を通じてこの整列を達成するためのメソッドを提供します。
